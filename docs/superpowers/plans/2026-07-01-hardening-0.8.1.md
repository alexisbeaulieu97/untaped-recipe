### Task 1: Encoding and newline discipline

Context: `src/` contains zero `encoding=`/`newline=` arguments. Consequences: (a) any
CRLF file is silently rewritten to LF by a transform round-trip (POSIX included); (b)
`backup.py` hashes in-memory text (`_hash_text`, utf-8) but verifies against on-disk
bytes written through locale/newline translation, so restore can spuriously demand
`--force`; (c) a hook emitting non-UTF-8 bytes kills a drain thread with a silent
`UnicodeDecodeError`, after which requests block until timeout.

Policy, applied uniformly:
- Every `read_text`/`write_text`/`open` of file CONTENT (target files, staged temp
  files, backup blobs, template/copy sources, `--content-file`) uses
  `encoding="utf-8", newline=""` on read AND write — line endings ride through
  byte-identically, and on-disk bytes equal `text.encode("utf-8")` so text hashes and
  byte hashes agree on every platform.
- Every read of METADATA text (recipe YAML, args/inputs fixture files) uses
  `encoding="utf-8"` (newline translation is harmless for parsed formats;
  `tomllib` paths already read bytes and are out of scope).
- The worker `Popen` gets `encoding="utf-8", errors="replace"` (Popen applies one
  codec to all streams); garbled protocol lines then fail JSON/Pydantic validation
  loudly through the existing invalid-response path instead of killing the drain
  thread. `hook_worker.py` mirrors it with
  `sys.stdin.reconfigure(encoding="utf-8", errors="replace")` and the same for
  stdout/stderr at startup.

**Files:**
- Modify: `src/untaped_recipe/infrastructure/file_writer.py` (staging write ~:77-78, verify read ~:59, restore read ~:122), `src/untaped_recipe/infrastructure/backup.py` (stage ~:70-73, restore ~:159, `_current_hash` stays byte-based — now consistent), `src/untaped_recipe/application/apply_recipe.py` (target reads ~:153, ~:168), `src/untaped_recipe/application/run_hook.py` (content reads), `src/untaped_recipe/infrastructure/hook_worker_client.py` (`Popen` kwargs ~:377-394), `src/untaped_recipe/hook_worker.py` (stream reconfigure at startup), plus any `read_text`/`write_text` the Step 1 grep finds

### Task 2: Worker process lifecycle

Context: `Popen` lacks `start_new_session=True`, so kill-escalation
(`hook_worker_client.py` `close()` ~:352-364) signals only the `uv` process — a hung
hook (the grandchild `python hook_worker.py`) can survive a timeout. And the pool and
group `close()` loops (~:116-117, ~:187-188) have no per-worker `try/except`, so one
`BrokenPipeError` from a dead worker's stdin aborts cleanup of every remaining
worker, on exactly the SIGINT/error path where cleanup matters most.

**Files:**
- Modify: `src/untaped_recipe/infrastructure/hook_worker_client.py`
- Test: `tests/test_hook_worker_client.py` (or wherever the pool tests live — `grep -rl "UvHookWorkerPool" tests/`)

**Interfaces (behavioral contract):**
- Spawn: `start_new_session=True` (POSIX; harmless no-op guard on Windows).
- Worker `close()`: `stdin.close()` wrapped in `except OSError: pass`; escalation `SIGTERM → wait(2) → SIGKILL → wait(2)` delivered via `os.killpg(os.getpgid(pid), sig)` when process groups are available, falling back to `terminate()`/`kill()` (and swallowing `ProcessLookupError` — the group may already be gone).
- Pool/group `close()`: each `worker.close()` in `try/except Exception`; all workers get a close attempt; the first captured exception is re-raised after the loop (others suppressed as `__context__` chaining is fine).

### Task 3: Input and stdin-target strictness

Context: (a) a CLI `--input-from` expression shadows the recipe's own `from` and, when
it renders Jinja-Undefined for a target (`inputs.py` `_resolve_one` ~:161-167), falls
silently through to default/required-error — an explicit override that fails should be
an error; (b) a stdin line that is valid non-object JSON (`42`, `[1]`, `"x"`) passes
the envelope check as False and becomes `Target(path=Path("42"))`
(`targets.py` ~:26-35); (c) `optional: true` transform after an in-recipe `remove`
raising `"cannot transform deleted file"` is CORRECT (the recipe contradicts itself)
but untested — pin it.

**Files:**
- Modify: `src/untaped_recipe/application/inputs.py`, `src/untaped_recipe/application/targets.py`
- Test: `tests/test_inputs.py`, `tests/test_targets.py`, apply behavior tests (pin test)

**Interfaces:**
- `_resolve_one`: when the source came from `config.cli_sources` and `_derive_source_value` returns `UNRESOLVED`, raise `ValueError(f"--input-from for input {name!r} did not resolve for target: {target.path}")`. Recipe-declared `from` keeps today's fall-through (deriving from an absent `record` field and landing on the default is by design).
- `resolve_target_lines`: a line that parses as JSON but is not an object and not an envelope raises `ValueError(f"stdin line {lineno} is not a pipe record or a path: {line!r}")`. Non-JSON lines remain bare paths, envelope lines unchanged.
